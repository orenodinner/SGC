# 10 Agent Autonomy Policy

## 目的
Codex が毎回ユーザーに確認せず、長期間実装を継続できるようにする。

## 原則
- 判断コストはユーザーではなくドキュメントへ移す
- 質問の代わりに既定値を増やす
- 既定値で危険なものだけ人間に戻す

## 確認しない既定値

### プロダクト既定
- UI 言語: 日本語
- 週開始: 月曜
- FY 開始月: 4
- 稼働日: 月〜金
- データ保存: SQLite ローカル
- 表示テーマ: ライト基調
- 主要ビュー初期表示: Home / Today

### 計画既定
- 日付1個のみ入力: 1日タスク
- 開始日なし + 期限のみ: due_date のみ保持、scheduled=true にはしない
- 日付範囲入力: scheduled=true
- milestone は開始=終了
- 親日付は子から導出
- 親進捗は子からロールアップ

### 依存既定
- 依存種別は FS のみ
- lag の単位は日
- 自動シフトは project detail 上の操作でのみ適用
- 循環依存は常に禁止

### Excel 既定
- canonical sheet は `Tasks`
- 文字コード問題は UTF-8 相当維持を最優先
- import エラー行はスキップして一覧化
- 同名 project がある場合は code 優先、無ければ name 一致で再利用

### UI 既定
- 右クリックメニューは最小
- destructive 操作のみ確認
- 保存ボタンはできる限り排除
- 高度設定は折りたたむ

## ユーザーに聞かずに選ぶべき技術判断
- 小規模 helper の作成
- コンポーネント分割粒度
- internal DTO の形
- directory 構成の微修正
- テストライブラリ内の fixture の持ち方
- virtualized list 実装の具体的方法

## 質問が許される条件
- 認証情報が必要
- 外部サービス利用が必要
- 既存データ移行で不可逆処理が必要
- 仕様書上の矛盾が重大
- 法的 / 契約的 / ブランド的判断
- 2案のどちらでも作れるが、片方でデータ互換性が壊れる

## Blocker でなければ行うこと
- 仮説を採用して前へ進む
- 仮説を `docs/decisions.yaml` に記録する
- 受け入れ試験へ影響があれば更新する

## 1セッションの完成条件
- 何かしら動くものが増えている
- テストが増えるか更新されている
- 状態ファイルが更新されている
- 次の切れ目が明記されている

## 優先順位の規則
1. 壊れた build/test を直す
2. P0 の未完了を進める
3. P1 の依存ロジックと Excel round-trip
4. P2 の快適機能
5. 見た目だけの改善は後回し

## 変更幅の規則
- 1回で repo 全体を書き換えない
- 仕様外の sweeping refactor をしない
- バグ修正は関連箇所まで広げるが、別件まで持ち込まない

## 迷った時の tie-breaker
- 実装の早さより再開のしやすさ
- 見た目より操作数の少なさ
- 汎用性よりこのプロダクトの主要ユースケース
- 新規抽象化より明快さ
